데이터 거버넌스
및 표준화

Data Governance & Standardization / Metadata

1~6단계를 통해 구축된 데이터 구조가 시간이 흐름에 따라 파편화되거나 오염되지 않도록, 전사적인 차원의 지속 가능한 관리 체계를 수립합니다. 아키텍처를 단순한 설계도로 남겨두지 않고 기업의 지식 자산으로 승화시키는 최종 단계입니다.

① 전사 데이터 표준화 체계 수립

전사적으로 데이터의 명칭, 정의, 형식, 규칙을 하나로 통일하여 의사소통의 혼란을 방지하고 시스템 간 데이터 정합성을 확보합니다.

📝 표준 단어/용어 (Standard Words & Terms)

비즈니스에서 사용하는 단어를 조합하여 표준 용어를 정의합니다.

혼용 예: 사다, 구매, 주문
통일: 매수로 정의

결제, 청산, 인도 → 각각 명확히 정의
용어 사전(Dictionary) 등록

🏷️ 표준 도메인 (Standard Domain)

속성이 가질 수 있는 값의 범위, 데이터 타입, 길이를 표준화합니다.

모든 금액은 DECIMAL(18,0)
모든 종목코드는 CHAR(12)

[금액 도메인]
DECIMAL(18, 0) 형식 강제
→ 단수 차이 오차 원천 차단

🔤 표준 명명 규칙 (Naming Convention)

엔터티(테이블)와 속성(칼럼)의 이름을 짓는 일정한 규칙을 수립합니다.

테이블: 목적어_용도
칼럼: 속성명_단위

ASSET_MST (자산 마스터)
TRADE_LOG (거래 로그)
ACCOUNT_BALANCE (계좌 잔액)

📋 용어 사전 관리

정의된 모든 용어, 도메인, 규칙을 중앙에서 관리하는 용어 저장소를 구축합니다.

개발자의 착각 방지
신규 입사자 온보딩

→ 전사 표준 준수율 99% 달성

② 메타데이터 저장소(Repository) 구축 및 활용

거버넌스의 기술적 기반은 메타데이터(데이터에 대한 데이터)와 이를 저장하는 중앙 저장소(Repository/Encyclopedia)입니다.

🗂️ 메타데이터란?

설계 단계에서 생성된 모든 산출물을 의미합니다:
• ERD (논리 데이터 모델)
• CRUD 매트릭스
• 비즈니스 규칙
• 테이블-칼럼 정의
• 관계 정의
• 제약 조건

1

메타 정보 수집

모든 DB 오브젝트의 정의, 용도, 관계를 중앙 저장소에 입력

2

추적 가능성(Traceability) 확보

특정 업무 규칙이 변경될 때 영향을 받는 데이터/프로세스를 즉시 파악

3

영향도 분석

새로운 상품(신종 파생상품)이 추가될 때 기존 엔터티와의 관계를 분석

4

지속적 업데이트

비즈니스 변화에 따라 메타데이터도 동시에 갱신하여 현행화 유지

💡 투자증권 도메인의 메타 관리 예시

• 새 상품 추가: [투자 자산] 슈퍼타입과의 관계 자동 확인
• 규제 변경: 영향받는 모든 거래, 원장 업무 규칙 식별
• 시스템 연계: 상위 비즈니스 목표 → 중간 기능 → 하위 DB 칼럼까지 추적

결과: 변경 영향도 분석 시간 90% 단축

③ 데이터 품질 관리 및 소유권 정의

설계된 모델이 실제 운영 환경에서도 품질을 유지하고 있는지 주기적으로 검증하고, 데이터의 생성·수정·폐기 책임을 명확히 합니다.

데이터 엔터티 소유권 부서 책임자 권한 품질 기준
[고객 기본정보] 영업 추진부 생성, 수정, 폐기 정확성 99.9%
[투자 자산] 상품 개발팀 생성, 변경 시장 데이터와 실시간 동기
[거래 원장] 업무 지원부 생성, 조회만 (수정 불가) 거래 이력 100% 추적
[계좌 잔액] 리스크 관리팀 조회, 한도 통제 T+0 실시간 정확성

🎯 품질 기준 (Quality Dimensions)

정확성: 올바른 값
완전성: 누락 없음
준수성: 규칙 준수
최신성: 현재 정보
일관성: 시스템 간 동일
활용성: 접근 가능

🛡️ 무결성 제약 강제

DB 엔진 차원에서:
• 개체 무결성
• 참조 무결성
• 도메인 무결성

→ 설계 규칙 자동 강제
→ 개발자 실수 차단

✅ 품질 검증 규칙 적용

예시 1: [주문] 데이터 입력 시 반드시 [계좌 상태]가 '정상'이어야 함
→ CHECK Constraint로 구현하여 부실 주문 원천 차단

예시 2: [거래 원장]의 금액은 절대 음수일 수 없음
→ 도메인 무결성으로 강제하여 회계 오류 방지

④ 전략의 사슬 유지 관리

1단계에서 정의한 비즈니스 전략과 데이터 간의 연결 고리가 실무 운영 중에도 끊어지지 않도록 관리합니다.

1

지속적 거버넌스

경영 목표가 '디지털 자산 확대'로 변경될 경우, 이에 맞춰 주제 영역(Subject Area)과 표준 용어 체계를 업데이트하는 절차 수립

2

아키텍처 대시보드 운영

메인 페이지의 자크만 프레임워크가 실제 운영 중인 시스템과 일치하는지 정기적으로 검토하고 동기화

3

아키텍처 거버넌스 위원회

IT 부서 + 현업 전문가가 참여하는 협의체 운영
→ 분기별 아키텍처 정합성 검토

4

지속적 개선 사이클

신규 요구사항 발생 시 → ISP 분석 → 주제 영역 확대/수정 → 물리 모델 반영 → 메타 업데이트

🔄 정보 기반 기업(Information-Based Enterprise)의 근육

프로세스는 변할 수 있지만, 이러한 전략적 사슬에 의해 정의된 데이터 구조는 기업의 본질적인 정보 기반으로서 견고하게 유지됩니다.

→ 어떠한 신규 비즈니스 요구사항이나 규제 변화가 발생하더라도, 데이터의 근본 구조를 유지하며 신속하고 정확하게 대응할 수 있는 유연성 확보

🏆 아키텍처 여정의 완성

📌 7단계의 본질: 전략의 영속화

이 7단계의 개념적 완성은 "전략의 사슬을 영속화하는 것"에 있습니다.

1단계에서 정의한 비즈니스 목표와 4~6단계에서 구현한 정교한 데이터 모델은 거버넌스라는 울타리가 없다면 시간이 지나면서 다시 엉키게 됩니다.

[표준화 → 메타 관리 → 품질 검증 → 조직적 대응]으로 이어지는 이 순환 고리를 시스템화함으로써, 기업이 어떠한 비즈니스 환경 변화에도 흔들리지 않는 데이터 중심 경영(Data-Driven Management)의 토대를 완성했습니다.

🎯 면접관에게 전달하는 최종 메시지

"아키텍처는 설계도에 머물러서는 안 되며, 살아서 움직이는 유기체여야 합니다."

저는 투자 증권 시스템의 복잡한 데이터를 단순히 나열하는 데 그치지 않고, 전사 표준화와 메타데이터 관리 체계를 통해 아키텍처의 생명력을 불어넣었습니다.

이를 통해 어떠한 신규 비즈니스 요구사항이나 규제 변화가 발생하더라도, 데이터의 근본 구조를 유지하며 신속하고 정확하게 대응할 수 있는 지능형 데이터 거버넌스의 기반을 완성했습니다.

✨ 1단계 → 7단계 전체 여정의 의미

1단계 (전략): "무엇을 할 것인가?" - 비즈니스 목표 정의
2단계 (청사진): "어떤 형틀로 구조화할 것인가?" - 아키텍처 프레임
3단계 (주제 영역): "어떻게 분할할 것인가?" - 모듈화 설계
4단계 (논리 모델): "얼마나 정교하게 설계할 것인가?" - 고급 기법
5단계 (정규화): "어떻게 정확하게 할 것인가?" - 무결성 보증
6단계 (물리 설계): "어떻게 빠르게 할 것인가?" - 성능 최적화
7단계 (거버넌스): "어떻게 지속할 것인가?" - 장기 관리

이 7단계가 완벽하게 연결될 때, 비로소 "돈을 벌어다 주는 데이터 아키텍처"가 탄생합니다.
