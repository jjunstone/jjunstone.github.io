고난도 논리 데이터 모델링

Advanced ERD – 엔터티 관계의 복잡성 설계

3단계까지 CRUD 정합성이 검증된 12개 엔터티를 이제 정교한 관계 모델로 표현합니다. 일반화 구조(슈퍼타입/서브타입), 재귀적 관계, 시계열 선분 이력 등 고난도 패턴을 적용하여 엔터피스 수준의 데이터 무결성을 보장합니다.

ERD 설계의 핵심 원칙

① 정규화 (Normalization)

엔터티의 속성을 함수 종속성(FD)에 따라 분해하여 데이터 중복을 제거하고 갱신 이상(Anomaly)을 방지합니다.

예: ORDER와 ORDER_STATUS를 분리하여 주문 상태 변경 시에만 ORDER_STATUS 업데이트

② 엔터피 무결성 (Entity Integrity)

모든 엔터티는 고유 식별자(PK)를 가지며, PK 속성은 NULL이 될 수 없습니다.

예: ORDER.ORDER_ID (PK), EXECUTION.EXEC_ID (PK)

③ 참조 무결성 (Referential Integrity)

FK는 참조하는 테이블의 PK 값만 가지거나 NULL이며, 고아 레코드(Orphan Record) 발생을 방지합니다.

예: EXECUTION.ORDER_ID → ORDER.ORDER_ID (ON DELETE CASCADE)

④ 도메인 무결성 (Domain Integrity)

속성 값이 정의된 도메인(데이터 타입, 범위, 형식)을 따릅니다.

예: RISK_LIMIT.LIMIT_VALUE ≥ 0, ORDER.STATUS ∈ {PENDING, EXECUTED, CANCELLED}

ERD의 고난도 패턴

아래 네 가지 패턴은 단순한 1:N, M:N 관계를 넘어, 비즈니스 로직의 복잡성을 데이터 모델에 정확히 반영합니다.

패턴 1️⃣ 슈퍼타입/서브타입 (Supertype/Subtype)
ASSET (PK: ASSET_ID)
▼

적용: ASSET_CLASS (주식/채권/파생상품)에 따라 속성이 달라지므로 슈퍼타입(ASSET) + 서브타입(STOCK, BOND, DERIVATIVE) 구조.
장점: 공통 속성(이름, 통화)은 한 번만 저장, 타입별 속성은 서브타입에서 관리.
관계: ASSET 1 ── N STOCK/BOND/DERIVATIVE (is-a 관계)

패턴 2️⃣ 재귀적 관계 (Recursive Relationship)
ASSET (ASSET_ID, PARENT_ASSET_ID)

적용: ASSET_CLASS가 계층(Level 1: 주식, Level 2: 국내주식, Level 3: KOSPI 주식)을 이룬다면 ASSET_CLASS 자신이 자신을 참조.
실제: ASSET_CLASS.PARENT_CLASS_ID FK → ASSET_CLASS.CLASS_ID
쿼리: "KOSPI 주식의 모든 상품" = ASSET JOIN ASSET_CLASS (재귀 검색)

패턴 3️⃣ 선분 이력 관리 (Slowly Changing Dimension, SCD Type 2)
PRICE_HISTORY
(ASSET_ID, PRICE_DATE, VALID_FROM, VALID_TO)

적용: 상품의 가격이 시간에 따라 변할 때, 새로운 가격이 발생하면 이전 행의 VALID_TO를 업데이트하고, 새 행을 삽입.
예:
   (ASSET_001, 2024-01-01, 2024-01-01, 2024-01-15) → PRICE = 100,000
   (ASSET_001, 2024-01-16, 2024-01-16, 9999-12-31) → PRICE = 105,000
쿼리: "2024-01-10의 상품 가격" = WHERE VALID_FROM ≤ '2024-01-10' AND VALID_TO ≥ '2024-01-10'

패턴 4️⃣ 연결 엔터티 & 복합 PK (Composite Primary Key)
ORDER (ORDER_ID)
M ─── N
EXECUTION (EXEC_ID)

관계: 한 주문(ORDER)이 여러 체결(EXECUTION)을 가질 수 있고, 한 체결이 여러 정산(SETTLEMENT)과 연결될 수 있음.
모델: EXECUTION ( EXEC_ID, ORDER_ID FK, QUANTITY, PRICE, ...)

🔍 이 4가지 패턴의 교집합

실제 설계에서는 이 패턴들이 결합됩니다. 예: ASSET (슈퍼타입) → STOCK (서브타입) → PRICE_HISTORY (선분이력) → ORDER → EXECUTION (M:N)

정규화 수준 & 데이터 무결성 검증

정규화 수준 규칙 우리 설계의 적용
1NF (First) 원자값(Atomic Value)만 허용 모든 속성은 원자값 (예: ORDER_QTY는 숫자, 배열 ✗)
2NF (Second) 부분 함수 종속성 제거 ORDER_STATUS를 분리하여 ORDER의 상태 이력과 분리
3NF (Third) 이행 함수 종속성 제거 ASSET.CLASS_ID → ASSET_CLASS (이행 제거)
BCNF 모든 결정자가 후보 키 복합 PK 검증: (ASSET_ID, PRICE_DATE) ✓

✅ 우리의 데이터 무결성 전략

3NF를 기본 기준으로 삼으며, 성능과 비즈니스 요구를 위해 필요한 부분에서만 역정규화(Denormalization)를 적용합니다. 예: P&L 캐시 테이블은 조회 성능을 위해 역정규화, 하지만 실시간 갱신 트리거로 데이터 신뢰성 유지.

주요 엔터티 간 관계 요약

ASSET → MARKET_DATA: 1:N (상품 1개는 여러 시장정보)
ASSET ↔ ASSET_CLASS: N:1 (상품은 정확히 1개 분류)
ORDER → EXECUTION: 1:N (주문 1개는 여러 체결)
EXECUTION → SETTLEMENT: 1:N (체결 1개는 여러 정산)
EXECUTION → POSITION: N:1 (체결이 포지션을 생성/변경)
POSITION ↔ RISK_LIMIT: N:1 (포지션 여러개가 1개 한도와 연결)
EXECUTION → LEDGER: N:1 (여러 체결이 원장 1행으로 기록)
EXECUTION → TRANSACTION_LOG: N:1 (체결이 로그에 기록)

📌 4단계 정리

ERD를 통해 우리는 12개 엔터티의 관계를 정규화, 무결성 제약, 고난도 패턴으로 정확히 정의했습니다. 이는 애플리케이션 계층에서 비즈니스 로직 오류 방지(예: 미정산 주문 수정 불가)의 기초가 됩니다.
