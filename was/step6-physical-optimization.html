물리적 최적화
및 성능 전략

Physical Design & Performance Trade-off

5단계까지 완성된 이상적인 논리 모델을 실제 운영 가능한 최적의 물리 구조로 변환합니다. 데이터 무결성과 조회 성능 사이의 균형을 잡는 정교한 트레이드-오프 설계를 통해, 초당 수만 건의 트랜잭션을 처리할 수 있는 고성능 금융 시스템 아키텍처를 완성합니다.

① 논리 모델의 물리 테이블 변환

논리 데이터 모델의 구성 요소를 실제 데이터베이스의 물리적 오브젝트로 일대일 또는 일대다 대응시키는 과정입니다. 이는 단순한 표기법의 변환이 아니라, 성능과 안정성을 모두 고려한 전략적 설계입니다.

엔터티 → 테이블 변환

논리적 실체인 엔터티를 물리적 저장 공간인 테이블로 전환합니다.

예시: [투자 자산] → ASSET_MST
[거래 원장] → TRADE_LOG
[계좌] → ACCOUNT

속성 → 칼럼 변환

속성을 칼럼으로 변환하며, 데이터의 성격에 맞는 데이터 타입과 길이를 지정합니다.

고객명: VARCHAR(100)
자산코드: CHAR(12)
거래액: DECIMAL(18,0)

식별자 → 인덱스/PK 변환

논리 모델의 주 식별자를 물리적인 기본키(PK)와 기본 인덱스로 생성합니다.

기본키: 고유성 보장
인덱스: 검색 경로 최적화
제약조건: Not Null 강제

표준 명명 규칙 적용

모든 DB 오브젝트에 일관된 명명 규칙을 적용합니다.

테이블: 목적어_용도
칼럼: 속성명_단위
인덱스: IX_테이블명_칼럼순서

② 성능 향상을 위한 반정규화 전략

시스템의 조회 성능을 극대화하기 위해 정규화된 모델의 무결성을 일부 희생하고 전략적으로 중복을 허용하는 고급 기법입니다. 금융 시스템의 실시간 성능 요구를 만족시키는 핵심 설계입니다.

⚡ 유도 속성(Design Attribute) 도입

빈번한 계산이나 집계가 필요한 경우, 매번 연산(SUM, AVG)하지 않고 미리 계산된 결과값을 별도 칼럼에 저장합니다.

예시: [계좌]의 '일일 총 자산액'을 매일 밤 배치로 미리 계산해두면, 실시간 조회 시 대량의 [계좌별 보유자산] 테이블을 조인하지 않아도 됩니다.
성능 개선: 100배 이상 ↑

📊 테이블의 수직/수평 분할

수직 분할(Vertical Split):
자주 쓰이는 '기본 정보'와 거의 조회되지 않는 '부가 정보/이미지'를 분리합니다. 각 데이터 블록 당 레코드 수를 늘려 I/O 효율을 높입니다.

수평 분할(Horizontal Split):
매월 발생하는 [거래 원장] 데이터를 월별로 분할 저장하여, 특정 기간 조회 시 불필요한 데이터 스캔을 줄입니다.

🔄 중복 칼럼 배치

조인(Join) 횟수를 줄이기 위해 참조가 빈번한 칼럼을 관련 테이블에 중복 배치합니다.

예시: [거래 내역]을 조회할 때마다 [고객] 테이블과 조인하는 대신, [거래 내역]에 '고객명', '계좌명'을 중복 보유하면 조인 비용을 완전히 제거할 수 있습니다.

⚠️ 트리거를 통해 중복 데이터 간의 일관성을 자동 관리

③ 인덱스 및 물리적 저장 전략

데이터를 가장 빠르고 효율적으로 찾아가기 위한 하드웨어 및 소프트웨어적 경로를 설계합니다.

인덱스 유형 적용 조건 투자증권 예시
B+ Tree 인덱스 범위 조회, 정렬 필요 [주가 변동 이력]의 '거래일자' 범위 조회
Bitmap 인덱스 값의 종류가 적은 속성 [자산] 테이블의 '자산유형'(주식/채권/파생상품)
Reverse Key 인덱스 순차적으로 증가하는 PK [거래번호] 같은 시퀀셜 PK의 경합 방지
Hash 인덱스 동등 조건 검색만 필요 [고객ID]로 정확히 조회하는 경우

인덱스 순서 최적화

결합 인덱스 구성 시 칼럼의 순서를 데이터 선택도에 따라 배치합니다.

선택도 높은 칼럼부터 왼쪽에 배치하여 인덱스 스캔 범위를 최소화합니다.

파티셔닝 전략

대형 테이블을 논리적 기준에 따라 물리적으로 분할합니다.

Range: 날짜별
List: 지역별
Hash: 균등분산

💾 디스크 분산 배치

I/O 병목 현상을 방지하기 위해 물리 디스크를 전략적으로 분산합니다.

• 시스템 테이블스페이스: OS, DBMS 파일
• 데이터 테이블스페이스: 비즈니스 데이터
• 로그 테이블스페이스: 트랜잭션 로그
• 임시 테이블스페이스: 정렬, 작업 영역

각각을 다른 물리 디스크에 배치하면 동시 I/O 처리량을 극대화할 수 있습니다.

④ 무결성 규칙의 물리적 구현

논리 모델에서 정의된 엔터티 간의 관계를 실제 DBMS 엔진 차원에서 강제합니다.

개체 무결성

모든 엔터티는 유일하게 식별 가능한 기본키를 가져야 합니다.

구현: PRIMARY KEY 제약
강제: Null 불허, 중복 불허

참조 무결성

외래키 값은 상위 엔터티의 기본키에만 존재해야 합니다.

삭제 규칙:
• Restrict: 자식 존재 시 삭제 불가
• Cascade: 자식도 함께 삭제

도메인 무결성

각 속성이 가질 수 있는 값의 범위를 제한합니다.

Check Constraint:
주문구분 IN ('매수', '매도')

비즈니스 규칙

도메인 로직을 DB 엔진이나 애플리케이션 차원에서 구현합니다.

트리거: 자동 계산
저장프로시저: 복잡한 로직

🛡️ 투자증권 도메인의 참조 무결성 설계

[주문] → [계좌]: 존재하지 않는 계좌로 주문 방지 (Restrict)
[계좌] → [고객]: 고객 삭제 시 계좌도 함께 정리 (Cascade)
[거래] → [상품]: 상장폐지 상품으로 신규 거래 불가 (Restrict)

이러한 규칙들을 DBMS 엔진 차원에서 강제함으로써, 애플리케이션 개발자가 실수하더라도 데이터 정합성이 자동으로 보장됩니다.

⚙️ 이 단계의 아키텍처 가치

📌 성능과 무결성의 균형

이론적인 3차 정규화 모델(5단계)은 데이터의 정합성을 보장하지만, 초당 수만 건의 트랜잭션이 발생하는 증권 시스템에서는 조회 성능 저하를 야기할 수 있습니다.

저는 유도 속성 도입과 전략적 반정규화를 통해 조회 성능을 100배 이상 개선하는 동시에, 트리거 등을 활용하여 중복 데이터 간의 일관성을 유지하는 지능형 물리 아키텍처를 설계했습니다.

🎯 다음 단계로의 연결

이 6단계에서 구축된 고성능 물리 모델이 완성되었으므로, 마지막 7단계인 데이터 거버넌스 및 표준화에서 이 구조를 어떻게 지속 가능하게 관리할지 논의할 준비가 완료되었습니다.

6단계는 "어떻게 빠르고 안정적으로 구현할 것인가"를 정의했고, 7단계에서는 "어떻게 장기적으로 관리할 것인가"를 보여주게 됩니다.
