PART A — 논리 ERD

ERD 설계의 핵심 원칙

ERD는 테이블 목록이 아니다. 비즈니스 규칙이 데이터 구조 안에 어떻게 내재되는지를 보여주는 설계 언어다. 카드를 클릭하면 상세 설명이 펼쳐진다.

① 정규화 (Normalization)

함수 종속성(FD)에 따라 속성을 분해 — 데이터 중복 제거 및 갱신 이상 방지

ORDER와 ORDER_STATUS를 분리 → 주문 상태 변경 시 ORDER_STATUS만 갱신. 주문 기본 정보는 불변 상태 유지.

② 엔터티 무결성 (Entity Integrity)

모든 엔터티는 고유 식별자(PK)를 가지며 PK는 NULL이 될 수 없음

ORDER.ORDER_ID (PK, NOT NULL), EXECUTION.EXEC_ID (PK, NOT NULL). PK 없는 엔터티는 존재를 증명할 수 없다 — "행"이 아니라 "노이즈"다.

③ 참조 무결성 (Referential Integrity)

FK는 참조 테이블의 PK에만 존재 — 고아 레코드(Orphan Record) 원천 차단

EXECUTION.ORDER_ID → ORDER.ORDER_ID (ON DELETE RESTRICT). 존재하지 않는 주문에 대한 체결 레코드 생성 불가. DB 엔진이 강제한다.

④ 도메인 무결성 (Domain Integrity)

속성 값이 정의된 도메인(타입·범위·형식)을 따름

RISK_LIMIT.LIMIT_VALUE ≥ 0 (CHECK), ORDER.STATUS IN ('PENDING','EXECUTED','CANCELLED'). 불가능한 값은 DB 엔진이 거절한다.

ERD의 고난도 패턴

단순 1:N을 넘어 비즈니스 로직의 복잡성을 모델에 정확히 담는 4가지 패턴. 탭을 클릭하면 전환된다.

① 슈퍼 / 서브타입
② 재귀적 관계
③ 선분 이력 SCD

자산 유형에 따라 필요한 속성이 완전히 다를 때

주식·채권·파생상품은 공통 속성(이름, 통화)은 같지만 유형별 속성(만기일, 이자율, 기초자산)이 다르다. 슈퍼타입이 공통을 담고, 서브타입이 유형별 확장을 담는다. 하나의 ASSET 테이블에 몰아넣으면 NULL 폭발이 발생한다.

-- 슈퍼타입 ASSET (ASSET_ID PK, CLASS_ID FK, ASSET_NAME, CURRENCY) -- 서브타입 (각각 FK로 ASSET 참조) STOCK (ASSET_ID FK PK, MARKET_CODE) BOND (ASSET_ID FK PK, MATURITY_DATE, COUPON_RATE, FACE_VALUE) DERIV (ASSET_ID FK PK, UNDERLYING_ID, EXPIRY_DATE, STRIKE_PRICE)

분류 체계가 계층 구조를 가질 때

자산 분류는 여러 계층을 이룬다(주식 → 국내주식 → KOSPI 주식). ASSET_CLASS가 자기 자신을 참조(PARENT_CLASS_ID → CLASS_ID)함으로써 무한 깊이의 계층을 하나의 테이블로 표현한다. 계층이 바뀌어도 테이블 구조는 변경할 필요가 없다.

ASSET_CLASS ( CLASS_ID PK, CLASS_NAME, CLASS_LEVEL, PARENT_CLASS_ID FK → CLASS_ID ) -- 계층 전체 조회 (재귀 CTE) WITH RECURSIVE hier AS ( SELECT * FROM ASSET_CLASS WHERE CLASS_ID = 'ROOT_STOCK' UNION ALL SELECT c.* FROM ASSET_CLASS c JOIN hier h ON c.PARENT_CLASS_ID = h.CLASS_ID ) SELECT * FROM hier;

쿼리 시점(01-10) → Row 1이 반환됨

데이터가 시간에 따라 변하고 과거 시점 복원이 필요할 때

가격이 변경될 때 기존 행을 수정하지 않는다. 이전 행의 VALID_TO를 닫고, 새 행을 삽입한다. 어떤 과거 시점의 가격도 WHERE 조건 하나로 정확하게 복원할 수 있다. 배치 마감 후 데이터 재현 요건의 핵심 패턴이다.

-- 선분 이력 구조 PRICE_HISTORY ( ASSET_ID PK, FK → ASSET, VALID_FROM PK, -- 복합 PK VALID_TO DEFAULT '9999-12-31', CLOSE_PRICE DECIMAL(18,0) NOT NULL ) -- 특정 시점 조회 (2024-01-10) SELECT CLOSE_PRICE FROM PRICE_HISTORY WHERE ASSET_ID = 'ASSET_001' AND VALID_FROM <= '2024-01-10' AND VALID_TO >= '2024-01-10'; -- → 100,000 반환 -- 현재 가격 WHERE VALID_TO = '9999-12-31';

주문 1개가 여러 체결로, 체결 1개가 여러 정산으로 이어질 때

대량 주문은 수십 번에 걸쳐 분할 체결된다. 각 체결은 별도의 정산 사이클을 가진다. 연결 엔터티(EXECUTION)가 ORDER와 SETTLEMENT 사이의 복잡한 관계를 명확히 분리하고, 각 단계의 책임을 독립적으로 관리한다.

ORDER -- 주문 (의도) ├── ORDER_ID PK EXECUTION -- 체결 (실행 결과) ├── EXEC_ID PK ├── ORDER_ID FK → ORDER ├── EXEC_QTY -- 이번 체결 수량 └── EXEC_PRICE SETTLEMENT -- 정산 (결제 완료) ├── SETL_ID PK ├── EXEC_ID FK → EXECUTION └── SETL_DATE

PART B — 정규화

정규화의 목표 — 이상 현상 제거

정규화란 데이터를 최적의 논리적 위치에 배치하여 정확성·일관성·비중복성을 확보하는 과정이다. 하지 않으면 세 가지 이상 현상이 생긴다.

🔴

삽입 이상 (Insertion Anomaly)

새 데이터 입력 시 불필요한 속성이 필수로 요구됨. 예: 고객 없이 상품만 등록하려 해도 고객 정보를 채워야 하는 상황.

🟡

수정 이상 (Update Anomaly)

데이터 수정 시 연관된 모든 행을 함께 수정해야 함. 예: 삼성전자 이름 변경 시 수백만 행의 거래 내역을 모두 수정해야 함.

🟢

삭제 이상 (Deletion Anomaly)

데이터 삭제 시 의도하지 않은 정보까지 손실됨. 예: 마지막 주문 삭제 시 고객 정보도 함께 사라지는 상황.

정규화 수준 요약 & 무결성 전략

수준핵심 규칙이 설계 적용
1NF원자값만 허용, 반복 그룹 제거모든 속성 단일 원자값 (ORDER_QTY는 숫자, 배열·콤마 분리 ✗)
2NF부분 함수 종속 제거ORDER_STATUS 분리 → 주문 상태 이력과 기본 정보 독립
3NF이행 함수 종속 제거ASSET.CLASS_ID → ASSET_CLASS 분리, 이행 종속 제거
BCNF모든 결정자가 후보키복합 PK (ASSET_ID, VALID_FROM) → 단일 의미 ✓

✅ 정규화 vs 역정규화 — 아키텍처적 균형

3NF를 기본 기준으로 삼는다. 단, 초당 수만 건 트랜잭션이 발생하는 증권 시스템에서는 전략적 역정규화를 적용한다.

예: P&L 캐시 — 매번 EXECUTION JOIN MARKET_DATA로 계산하는 대신 미리 계산된 손익을 저장. 조회 성능 대폭 확보. 트리거로 중복 데이터 간 일관성을 자동 유지한다.

주요 엔터티 간 관계 요약

관계유형설명

📌 이 단계가 정의한 것

논리 모델은 "데이터가 어떻게 생겼고 어떤 관계를 맺어야 하는가"를 정의했다. 정규화는 그 구조가 논리적으로 올바른지를 수학적으로 검증했다. 이 두 작업이 완료되어야 물리 설계가 의미 있다.

🎯 Step 5 — 물리 설계로의 연결

이상적인 논리 모델을 실제 DB 환경에 맞게 구현한다. 인덱스 전략, 파티셔닝, 반정규화, DDL 설계 — "어떻게 빠르고 안정적으로 구현할 것인가"를 다룬다.
