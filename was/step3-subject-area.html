Step 3 of 7

CRUD 매트릭스 분석

Data–Process Consistency Verification

2단계에서 분할된 4개 주제 영역의 엔터티들이 어떤 업무 기능에 의해 생성(C), 조회(R), 수정(U), 삭제(D)되는지 매핑합니다. 이를 통해 모든 데이터의 생성 소유권(Creation Ownership)을 검증하고, 기능–데이터 간 논리적 완결성을 증명합니다.

CRUD 분석의 본질

CRUD 매트릭스는 업무 기능(Function)과 데이터 엔터티(Entity) 간 상호작용을 4가지 연산으로 분류합니다. 아키텍처적으로 가장 중요한 원칙은 모든 엔터티는 반드시 하나 이상의 기능에 의해 생성(C)되어야 한다는 것입니다. 생성 주체가 없는 엔터티는 설계 오류를 의미합니다.

C
Create — 생성
데이터의 존재를 시작하는 행위. 소유권을 결정하는 가장 핵심적인 연산
R
Read — 조회
다른 기능의 의사결정을 위해 데이터를 읽는 행위. SA 간 의존성 파악의 핵심
U
Update — 수정
데이터의 상태를 갱신하는 행위. 이력 모델링(선분 이력)과 직결됨
D
Delete — 삭제
데이터 생명주기를 종료하는 행위. 금융 시스템에서는 논리 삭제가 원칙

✅ 검증 완료 — 12/12 엔터티 생성 소유권 확인

생성 주체가 없는 고아 엔터티가 없습니다. 모든 데이터는 명확한 비즈니스 기능에 의해 탄생하며, 이는 시스템의 데이터 완전성(Completeness)과 프로세스-데이터 정합성을 보증합니다.

데이터 생성 순서도 (Data Creation Flow)

CRUD 매트릭스의 C(Create) 연산을 업무 흐름 순서로 정렬하면, 데이터가 어떤 순서로 태어나는지 드러납니다. 이 순서가 업무 프로세스의 실행 순서와 일치할 때 아키텍처가 논리적으로 완결됩니다.

① 마스터 데이터 → ② 시세 데이터 → ③ 주문 → ④ 체결/포지션 → ⑤ 정산 → ⑥ 손익 → ⑦ 원장 (배치 마감)

클러스터링 분석 — SA별 데이터 소유권 지도

C(Create) 연산을 대각선 방향으로 정렬(Clustering)하면 독립적인 업무 모듈이 자연스럽게 분리됩니다. 어떤 업무 기능이 어느 SA의 데이터를 '소유'하는지 매핑합니다.

이 단계의 아키텍처 가치

📌 CRUD 분석이 증명하는 것

3단계는 "데이터가 어디에 있어야 하는가"를 넘어 "데이터가 어떻게 태어나고, 누가 그 생명에 책임지는가"를 정의하는 과정입니다. 생성 소유권의 명확화는 이후 데이터 거버넌스(7단계)의 기술적 근거가 되며, 시스템의 모듈화(Modularity) 수준을 수치로 증명합니다.

🎯 다음 단계로의 연결

CRUD 분석으로 데이터–프로세스 정합성이 검증되었습니다. 이제 4단계: 고난도 논리 데이터 모델링(Advanced ERD)에서 이 12개 엔터티들 사이의 복잡한 관계 — 일반화(슈퍼/서브타입), 재귀적 관계, 선분 이력 — 를 정교한 ERD로 표현합니다.
