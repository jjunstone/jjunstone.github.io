05

물리 설계 &
성능 전략

4단계까지 완성된 논리 모델과 정규화를 실제 운영 가능한 물리 구조로 변환한다. 데이터 무결성과 조회 성능 사이의 균형을 잡는 트레이드오프 설계를 통해, 초당 수만 건의 트랜잭션을 처리할 수 있는 고성능 금융 시스템 아키텍처를 완성한다.

논리 모델의 물리 변환

논리 모델의 구성 요소를 실제 DB의 물리 오브젝트로 대응시킨다. 단순한 표기법 변환이 아니라 성능과 안정성을 함께 고려하는 전략적 설계다. 행을 클릭하면 예시가 펼쳐진다.

엔터티→테이블 논리적 실체를 물리적 저장 공간으로
속성→칼럼 데이터 성격에 맞는 타입·길이 지정
식별자→PK / 인덱스 고유성 보장 + 검색 경로 최적화
관계→FK / 제약 참조 무결성을 엔진 차원에서 강제
표준 명명 규칙패턴예시
테이블목적어_용도ASSET_MST, TRADE_LOG
칼럼속성명_단위PRICE_KRW, QTY_SHR
인덱스IX_테이블명_칼럼순서IX_PRICE_HIST_ASSET_DATE

성능을 위한 반정규화 전략

조회 성능을 극대화하기 위해 정규화된 모델의 무결성을 일부 희생하고 전략적으로 중복을 허용하는 기법이다.

⚡

유도 속성 (Derived Attribute)

빈번한 계산·집계 결과를 미리 저장한다. 예: 계좌의 '일일 총 자산액'을 배치로 미리 계산해두면, 실시간 조회 시 대량의 보유자산 테이블을 매번 조인할 필요가 없다.

📊

테이블 수직·수평 분할

수직: 자주 쓰는 기본 정보와 거의 안 쓰는 부가 정보를 분리한다.
수평: 월별 거래 원장을 분할 저장해 특정 기간 조회 시 스캔 범위를 줄인다.

🔄

중복 칼럼 배치

조인 횟수를 줄이기 위해 참조가 빈번한 칼럼을 관련 테이블에 중복 배치한다. 예: 거래 내역에 고객명·계좌명을 중복 보유.

⚠️ 트리거로 중복 데이터 간 일관성을 자동 관리

유도 속성 적용 효과 — 일일 총 자산액 조회

실시간 계산
~320ms

인덱스 전략 탐색기

데이터의 특성에 따라 다른 인덱스 구조가 필요하다. 탭을 클릭해 각 인덱스가 내부적으로 어떻게 동작하는지 확인한다.

B+ Tree
Bitmap
Reverse Key
Hash

범위 조회·정렬이 필요할 때

트리 구조로 데이터를 정렬된 상태로 유지한다. 루트에서 리프까지 O(log n)으로 탐색하고, 리프 노드끼리 연결되어 있어 범위 조회를 순차적으로 빠르게 처리한다. 가장 범용적인 인덱스 구조다.

투자증권 예시: [주가 변동 이력]의 '거래일자' 범위 조회
WHERE TRADE_DATE BETWEEN '2024-01-01' AND '2024-03-31'
⚠️ 쓰면 안 되는 경우: 값의 종류가 매우 적은 컬럼 (예: 자산유형 3종류, 성별 2종류) — 이 경우 Bitmap 인덱스가 훨씬 효율적이다. 카디널리티가 낮을수록 B+Tree의 이점이 사라진다.

값의 종류가 적은(저카디널리티) 속성일 때

각 값마다 비트맵(0/1 배열)을 만들어 행의 존재 여부를 표시한다. 값의 종류가 적을수록 비트맵이 작고 빨라진다. B-Tree와 반대로 카디널리티가 낮을수록 유리하다.

투자증권 예시: [자산] 테이블의 '자산유형' (주식/채권/파생상품 — 단 3종류)
⚠️ 쓰면 안 되는 경우: 데이터가 자주 INSERT/UPDATE/DELETE되는 OLTP 환경 — Bitmap 재구성 비용이 높아 쓰기 성능이 급격히 저하된다. 실시간 거래 테이블에는 사용하지 않는다.

순차적으로 증가하는 PK를 동시에 많이 입력할 때

일반 순차 키는 항상 같은(최신) B-Tree 노드에 몰려 쓰기 경합(Hot Block)이 발생한다. 키의 비트 순서를 뒤집어 저장하면 해시처럼 흩어져 경합 없이 분산 입력된다.

투자증권 예시: [거래번호] 같은 시퀀셜 PK의 동시 입력 경합 방지
⚠️ 쓰면 안 되는 경우: 범위 조회(BETWEEN, >, <)가 필요한 컬럼 — Reverse 인덱스는 물리적 순서를 역전시켜 범위 검색 시 풀스캔이 발생한다. 거래일자 범위 조회에는 사용하지 않는다.

정확히 일치하는 값(동등 조건)만 검색할 때

키를 해시 함수에 통과시켜 정확히 하나의 버킷으로 직행한다. B-Tree처럼 여러 단계를 거치지 않고 O(1)에 가깝게 접근하지만, 범위 조회나 정렬에는 사용할 수 없다는 트레이드오프가 있다.

투자증권 예시: [고객ID]로 정확히 조회하는 경우
⚠️ 쓰면 안 되는 경우: 범위 조회·정렬(ORDER BY)·LIKE 연산이 필요한 경우 — Hash 인덱스는 정렬을 지원하지 않는다. 거래일자나 금액 범위 조회에는 B+Tree를 사용해야 한다.

파티셔닝 전략

대형 테이블을 논리적 기준에 따라 물리적으로 분할한다. 탭을 클릭해 분할 방식의 차이를 비교한다.

Range — 날짜별
List — 분류별
Hash — 균등분산

날짜·시간 기준으로 연속 구간을 나눌 때

월별 거래 원장을 분할 저장하면, 특정 기간 조회 시 해당 파티션만 스캔하여 불필요한 데이터 스캔을 줄인다. 오래된 파티션은 별도 보관(아카이빙)하기도 쉽다.

투자증권 예시: [거래 원장]을 월 단위로 분할 — 분기 결산 시 해당 분기 파티션만 접근

정해진 범주(카테고리)로 나눌 때

거래소·시장 구분처럼 미리 정의된 값 목록에 따라 분할한다. Range와 달리 값 사이에 순서나 연속성이 없고, 각 파티션이 독립적인 비즈니스 의미를 가진다.

투자증권 예시: [주문] 테이블을 거래소별(KOSPI/KOSDAQ/해외)로 분할 — 시장별 운영팀이 자기 파티션만 조회

데이터가 한쪽으로 쏠리지 않고 균등하게 퍼져야 할 때

해시 함수로 키를 계산해 여러 파티션에 고르게 분산한다. 논리적 의미는 없지만 I/O 부하를 가장 균등하게 나눠 병렬 처리 성능을 극대화한다.

투자증권 예시: [고객 계좌]를 고객ID 해시값으로 4개 파티션에 균등 분산 — 특정 우량 고객 데이터로의 쏠림 방지

무결성 규칙의 물리적 구현

논리 모델에서 정의된 관계를 DBMS 엔진 차원에서 강제한다. 카드를 클릭하면 실제 구현 문법이 펼쳐진다.

개체 무결성

모든 엔터티는 유일하게 식별 가능한 기본키를 가져야 한다.

-- ORDER 테이블 PK 정의
CONSTRAINT PK_ORDER
PRIMARY KEY (ORDER_ID)
-- ORDER_ID: NOT NULL + UNIQUE 자동 강제
-- 중복 주문 ID 원천 차단

참조 무결성

외래키 값은 상위 엔터티의 기본키에만 존재해야 한다.

-- EXECUTION의 참조 무결성
CONSTRAINT FK_EXEC_ORDER
FOREIGN KEY (ORDER_ID)
REFERENCES ORDER_MST(ORDER_ID)
ON DELETE RESTRICT
-- 존재하지 않는 주문의 체결 원천 차단

도메인 무결성

각 속성이 가질 수 있는 값의 범위를 제한한다.

CHECK (ORDER_TYPE
IN ('매수','매도'))

비즈니스 규칙

도메인 로직을 DB 엔진 차원에서 구현한다.

-- 반정규화 일관성 유지 트리거
CREATE TRIGGER trg_pos_upd
AFTER INSERT ON EXECUTION
FOR EACH ROW
BEGIN
UPDATE POSITION
SET QUANTITY = QUANTITY + NEW.EXEC_QTY,
MARKET_VALUE = ...
WHERE ASSET_ID = NEW.ASSET_ID;
END;

🛡️ 투자증권 도메인의 참조 무결성 설계

[주문] → [계좌]존재하지 않는 계좌로 주문 방지RESTRICT
[계좌] → [고객]고객 삭제 시 계좌도 함께 정리CASCADE
[거래] → [상품]상장폐지 상품으로 신규 거래 불가RESTRICT

물리 구현 — PRICE_HISTORY DDL

4단계에서 설계한 선분 이력(SCD Type 2) 패턴을 실제 DDL로 구현한 결과다. 개체·참조·도메인 무결성이 한 테이블 안에 모두 강제되어 있다.

-- 선분 이력: 자산 가격을 시점별로 추적
CREATE TABLE PRICE_HISTORY (
    ASSET_ID     CHAR(12)       NOT NULL,
    PRICE_DATE   DATE           NOT NULL,
    VALID_FROM   DATE           NOT NULL,
    VALID_TO     DATE           NOT NULL DEFAULT '9999-12-31',
    CLOSE_PRICE  DECIMAL(18,0)  NOT NULL,
    CURRENCY     CHAR(3)        NOT NULL DEFAULT 'KRW',

    CONSTRAINT PK_PRICE_HIST PRIMARY KEY (ASSET_ID, VALID_FROM),
    CONSTRAINT FK_PRICE_ASSET FOREIGN KEY (ASSET_ID)
        REFERENCES ASSET_MST(ASSET_ID),
    CONSTRAINT CK_VALID_RANGE CHECK (VALID_FROM <= VALID_TO),
    CONSTRAINT CK_PRICE_POS   CHECK (CLOSE_PRICE >= 0)
);

-- 인덱스: 자산별 최신가 조회 최적화 (B+ Tree)
CREATE INDEX IX_PRICE_HIST_ASSET_DATE
    ON PRICE_HISTORY (ASSET_ID, VALID_TO DESC);
💡 VALID_TO='9999-12-31'이 현재 유효 행을 의미한다. 가격이 바뀌면 기존 행을 UPDATE하는 대신 이전 행의 VALID_TO를 닫고 새 행을 INSERT — 물리 삭제·수정 없이 전체 이력이 보존된다.

📌 성능과 무결성의 균형

이론적으로 완성된 정규화 모델(4단계)은 데이터 정합성을 보장하지만, 초당 수만 건의 트랜잭션이 발생하는 증권 시스템에서는 조회 성능 저하를 야기할 수 있다. 유도 속성 도입과 전략적 반정규화로 조회 성능을 확보하는 동시에, 트리거로 중복 데이터 간 일관성을 유지하는 것이 이 단계의 핵심이다.

🎯 Step 6 — 거버넌스로의 연결

고성능 물리 모델이 완성되었으므로, 마지막 6단계인 데이터 거버넌스 및 표준화에서 이 구조를 어떻게 지속 가능하게 관리할지 다룬다. 5단계는 "어떻게 빠르고 안정적으로 구현할 것인가"를 정의했고, 6단계는 "어떻게 장기적으로 관리할 것인가"를 보여준다.
