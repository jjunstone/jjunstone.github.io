Physical Design & Performance Trade-off

4단계까지 완성된 이상적인 논리 모델을 실제 운영 가능한 최적의 물리 구조로 변환합니다. 데이터 무결성과 조회 성능 사이의 균형을 정교하게 조율하여, 초당 수만 건의 트랜잭션을 처리할 수 있는 고성능 금융 시스템을 완성합니다.

① 논리 모델의 물리 테이블 변환

논리 데이터 모델의 구성 요소를 실제 데이터베이스의 물리적 오브젝트로 매핑하는 과정입니다. 이는 단순한 표기법의 변환이 아니라, 성능과 안정성을 모두 고려한 전략적 설계입니다.

엔터티 → 테이블

논리적 실체를 물리적 저장 공간으로 전환합니다.

ASSET, EXECUTION, LEDGER 등 각 엔터티는 정확히 하나의 테이블이 됩니다.

속성 → 칼럼 + 타입

속성을 칼럼으로 변환하며, 데이터 성격에 맞는 타입과 길이를 지정합니다.

VARCHAR(100), DECIMAL(18,0), DATE, CHAR(12)

식별자 → 기본키 & 인덱스

주 식별자를 기본키로, 검색 경로를 인덱스로 구현합니다.

NOT NULL 강제, 유일성 보장, 접근 성능 최적화

표준 명명 규칙 적용

모든 DB 오브젝트에 일관된 명명을 적용합니다.

테이블: 목적어_용도
인덱스: IX_테이블_칼럼

② 성능 향상을 위한 반정규화 전략

조회 성능을 극대화하기 위해 정규화 원칙을 전략적으로 완화하고 목표하에 중복을 허용합니다. 금융 시스템의 실시간 성능 요구를 만족시키는 핵심 설계입니다.

⚡ 전략 1: 유도 속성(Derived Attribute) 도입

빈번한 계산이 필요한 경우, 매번 SUM/AVG를 실행하지 않고 미리 계산된 값을 별도 칼럼에 저장합니다.

예시: [계좌]의 '일일 총 자산액'을 매일 밤 배치로 미리 계산해두면, 실시간 조회 시 대량의 [계좌별 보유자산] 테이블을 조인하지 않아도 됩니다.
성능: 100배 이상 개선 ↑

📊 전략 2: 테이블의 수직/수평 분할

수직 분할(Vertical Split): 자주 쓰이는 '기본 정보'와 거의 조회되지 않는 '부가정보'를 분리합니다. 블록당 레코드 수를 늘려 I/O 효율을 극대화합니다.

수평 분할(Horizontal Split): 매월 발생하는 [거래 원장] 데이터를 월별로 분할 저장하여, 특정 기간 조회 시 불필요한 데이터 스캔을 줄입니다.

🔄 전략 3: 중복 칼럼 배치 (트리거로 일관성 유지)

조인(Join) 횟수를 줄이기 위해 참조가 빈번한 칼럼을 관련 테이블에 중복 배치합니다.

예시: [거래 내역]을 조회할 때마다 [고객] 테이블과 조인하는 대신, [거래 내역]에 '고객명', '계좌명'을 중복 보유하면 조인 비용을 완전히 제거합니다.
⚠️ 주의: 트리거(TRIGGER)로 중복 데이터 간 일관성을 자동 관리합니다.

③ 인덱스 전략 및 검색 경로 최적화

데이터를 가장 빠르고 효율적으로 찾아가기 위한 하드웨어 및 소프트웨어적 경로를 설계합니다.

인덱스 유형 적용 조건 투자증권 예시 성능 개선
B+ Tree 범위 조회, 정렬 필요 거래일자 범위 조회 (2024-01-01 ~ 2024-01-31) ~1000배
Bitmap 값의 종류가 적은 속성 자산유형 (주식/채권/파생상품) ~100배
Reverse Key 순차 증가 PK의 경합 방지 거래번호 시퀀셜 PK ~50배
Hash 동등 조건 검색만 필요 고객ID로 정확히 조회 (= 연산) ~500배

결합 인덱스 순서 최적화

SELECT에서 WHERE 조건과 일치하는 칼럼을 왼쪽부터 배치합니다.

INDEX(계좌ID, 거래일자) — 선택도 높은 것부터 배치

커버링 인덱스

인덱스만으로 쿼리를 완료할 수 있게 필요한 칼럼을 모두 포함합니다.

원본 테이블 접근 제거 → 성능 극대화

④ 파티셔닝 및 대용량 데이터 관리

대형 테이블을 논리적 기준에 따라 물리적으로 분할하여 관리와 성능을 동시에 확보합니다.

📅 Range 파티셔닝

날짜나 수치 범위로 분할합니다.

[TRADE_LOG]를 월별로 분할: TRADE_LOG_202401, TRADE_LOG_202402...

📍 List 파티셔닝

특정 값의 목록으로 분할합니다.

[계좌]를 지점별로 분할: ACCOUNT_Seoul, ACCOUNT_Busan...

🔀 Hash 파티셔닝

해시 함수로 균등 분산합니다.

고객ID를 4개 파티션에 해시로 분산: 균등한 데이터 분포 보장

🔗 Composite 파티셔닝

여러 기준을 조합합니다.

먼저 Range(월), 그 다음 List(지점)으로 2단계 분할

💡 파티셔닝의 효과

월별로 분할된 [TRADE_LOG]에서 "2024-01의 거래"를 조회하면, TRADE_LOG_202401 테이블만 스캔하고 나머지는 접근하지 않습니다. → I/O 비용 99% 감소, 조회 속도 100배 개선

⑤ 실제 DDL(Data Definition Language) 샘플

논리 모델을 물리 데이터베이스로 구현하는 실제 SQL 코드입니다. 선분 이력(SCD Type 2)과 무결성 제약을 포함합니다.

-- 선분 이력 예시: PRICE_HISTORY (SCD Type 2) CREATE TABLE PRICE_HISTORY ( ASSET_ID CHAR(12) NOT NULL, PRICE_DATE DATE NOT NULL, VALID_FROM DATE NOT NULL, VALID_TO DATE NOT NULL DEFAULT '9999-12-31', CLOSE_PRICE DECIMAL(18, 0) NOT NULL, CURRENCY CHAR(3) NOT NULL DEFAULT 'KRW', CONSTRAINT PK_PRICE_HIST PRIMARY KEY (ASSET_ID, VALID_FROM), CONSTRAINT FK_PRICE_ASSET FOREIGN KEY (ASSET_ID) REFERENCES ASSET_MST(ASSET_ID) ON DELETE RESTRICT, CONSTRAINT CK_VALID_RANGE CHECK (VALID_FROM <= VALID_TO), CONSTRAINT CK_PRICE_POS CHECK (CLOSE_PRICE > 0) ); -- 인덱스 생성 CREATE INDEX IX_PRICE_DATE ON PRICE_HISTORY(ASSET_ID, PRICE_DATE); CREATE INDEX IX_VALID_PERIOD ON PRICE_HISTORY(ASSET_ID, VALID_FROM, VALID_TO); -- 파티셔닝: 월별 Range ALTER TABLE PRICE_HISTORY PARTITION BY RANGE (YEAR_MONTH(PRICE_DATE)) ( PARTITION p202401 VALUES LESS THAN (202402), PARTITION p202402 VALUES LESS THAN (202403), PARTITION pmax VALUES LESS THAN MAXVALUE );

📝 코드 해석

VALID_FROM/VALID_TO: 선분 이력으로 특정 시점의 가격을 복원 가능
CHECK 제약: VALID_FROM ≤ VALID_TO, PRICE > 0를 DB 엔진이 강제
FOREIGN KEY ON DELETE RESTRICT: 상품 삭제 시 가격 이력이 있으면 방지
복합 PK: (ASSET_ID, VALID_FROM)로 각 자산의 각 시기를 유일하게 식별
파티셔닝: 월별로 자동 분할되어 조회 성능 극대화

⑥ 무결성 제약의 물리적 구현

논리 모델에서 정의된 엔터티 간 관계와 규칙을 DBMS 엔진 차원에서 강제합니다.

개체 무결성

모든 엔터티는 유일한 기본키를 가져야 합니다.

PRIMARY KEY 제약으로 Null 불허, 중복 불허 자동 강제

참조 무결성

외래키는 상위 엔터티 PK에만 존재합니다.

FOREIGN KEY 제약으로 고아 레코드(Orphan Record) 자동 방지

도메인 무결성

속성은 정의된 타입과 범위를 따릅니다.

CHECK CONSTRAINT로 값 범위 강제 (예: 주문구분 IN ('매수', '매도'))

비즈니스 규칙

복잡한 로직은 트리거/저장프로시저로 구현합니다.

예: 체결 시 포지션 자동 갱신, 거래액 자동 계산

🛡️ 금융 도메인 제약 예시

[주문] → [계좌]: Restrict — 존재하지 않는 계좌로 주문 방지
[계좌] → [고객]: Cascade — 고객 삭제 시 계좌도 함께 정리
[거래] → [상품]: Restrict — 상장폐지 상품으로 신규 거래 불가
[거래액]: CHECK(금액 ≥ 0) — 음수 금액 원천 차단

이러한 규칙들을 DBMS 엔진에서 강제함으로써, 애플리케이션 개발자가 실수하더라도 데이터 정합성이 자동으로 보장됩니다.

⑦ 디스크 분산 배치 및 I/O 최적화

I/O 병목 현상을 방지하기 위해 물리 디스크를 전략적으로 분산합니다. 초당 수만 건의 동시 트랜잭션을 처리하기 위한 필수 설계입니다.

🖥️ 시스템 테이블스페이스 (System Tablespace)
OS 파일, DBMS 카탈로그, 제어 파일
📊 데이터 테이블스페이스 (Data Tablespace)
비즈니스 데이터: ASSET, EXECUTION, LEDGER 등
📝 로그 테이블스페이스 (Log Tablespace)
트랜잭션 로그, REDO/UNDO 로그 (순차 쓰기용)
⚡ 임시 테이블스페이스 (Temp Tablespace)
정렬, 해시 조인, 그룹핑 등 작업 영역

최적 구성: 각 테이블스페이스를 별도의 물리 디스크(또는 RAID 그룹)에 배치합니다.

효과: 시스템 I/O가 데이터 조회를 블로킹하지 않고, 로그 쓰기가 데이터 읽기 경로와 겹치지 않아 동시성 극대화.

현실 예시: 초당 10,000건 거래 처리 시, 각 물리 디스크가 독립적으로 작동하면 처리량이 3~4배 증가합니다.

이 단계의 아키텍처 가치

📌 성능과 무결성의 균형

이론적인 3차 정규화 모델(Step 4)은 데이터 정합성을 보장하지만, 초당 수만 건의 트랜잭션이 발생하는 금융 시스템에서는 조회 성능 저하를 야기할 수 있습니다.

Step 5에서는 유도 속성 도입, 전략적 반정규화, 인덱스 최적화, 파티셔닝을 통해 조회 성능을 100배 이상 개선하는 동시에, 트리거와 제약조건으로 중복 데이터의 일관성을 자동 유지하는 지능형 물리 아키텍처를 설계합니다.

🎯 다음 단계로의 연결

이 5단계에서 구축된 고성능 물리 모델이 완성되었으므로, 마지막 6단계: 데이터 거버넌스 및 표준화에서 이 구조를 어떻게 지속 가능하게 관리할지 논의할 준비가 완료되었습니다.

5단계는 "어떻게 빠르고 안정적으로 구현할 것인가"를 정의했고, 6단계에서는 "어떻게 장기적으로 관리하고 진화시킬 것인가"를 보여줍니다.
