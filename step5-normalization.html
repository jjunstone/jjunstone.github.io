데이터 무결성 보장

Normalization – 정규화를 통한 이상 현상 제거

4단계까지 설계된 논리 모델을 이제 정규화 단계를 거쳐 최적의 논리적 위치에 배치합니다. 데이터 중복을 제거하고 함수적 종속성을 명확히 하여, 입출력 시 발생할 수 있는 이상 현상(Anomaly)을 체계적으로 차단합니다.

정규화의 목표와 이상 현상

정규화란? 데이터를 최적의 논리적 위치에 배치하여 데이터 구조의 정확성, 일관성, 단순성, 비중복성을 확보하는 과정입니다.

제거해야 할 이상 현상
• 삽입 이상: 새 데이터 입력 시 불필요한 속성이 필수로 요구되는 상황
• 수정 이상: 특정 데이터 수정 시 연관된 모든 행을 함께 수정해야 하는 상황
• 삭제 이상: 데이터 삭제 시 의도하지 않은 다른 정보까지 손실되는 상황

1단계: 제1정규형 (1NF) – 반복 그룹 제거

2단계: 제2정규형 (2NF) – 부분 함수 종속 제거

② 기본키 전체에 대한 완전 종속

❌ 문제: 복합 기본키 중 일부에만 종속되는 속성 존재

3단계: 제3정규형 (3NF) – 이행 함수 종속 제거

③ 기본키가 아닌 속성들 간의 종속성 제거

❌ 문제: 일반 속성들 사이에 종속 관계 존재 (지점명 → 지점주소)

④ BCNF (Boyce-Codd Normal Form) &ndash 선택사항

모든 결정자(Determinant)가 후보키가 되도록 하는 더 엄격한 정규화입니다. 실무에서는 3NF까지 일반적으로 적용하며, 특수한 경우에만 BCNF를 추가로 수행합니다.

정규화와 함께 강제되는 무결성 규칙

개체 무결성 (Entity Integrity)

모든 엔터티는 반드시 유일한 기본키(PK)를 가지며, 이 값은 NULL이 될 수 없습니다.

ORDER.ORDER_ID (PK, NOT NULL)

참조 무결성 (Referential Integrity)

외래키(FK) 값은 참조하는 테이블의 PK에 존재하거나 NULL이어야 합니다.

EXECUTION.ORDER_ID FK → ORDER.ORDER_ID

도메인 무결성 (Domain Integrity)

각 속성은 정의된 도메인(데이터 타입, 범위, 형식)을 따릅니다.

ORDER.STATUS ∈ {PENDING, EXECUTED, CANCELLED}

정규화의 영향 및 의의

📊 정규화 전후 비교

Before (비정규화): 테이블 적음, 조회 빠름, 하지만 중복 많고 수정 시 모순 발생 가능

After (정규화 완료): 테이블 많음, 조회 시 조인 필요, 하지만 중복 제거되고 무결성 강제됨

🎯 아키텍트의 통찰

정규화는 단순히 테이블을 쪼개는 기술이 아닙니다. 비즈니스 규칙을 데이터 구조 자체로 강제하는 과정으로, 이를 통해 애플리케이션 로직에 의존하지 않고도 데이터의 정합성을 체계적으로 보장할 수 있게 됩니다.

🔄 다음 단계로의 연결
